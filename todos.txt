OPNSENSE EXPORTER — METRICS COVERAGE IMPROVEMENT TODOS

This file contains independently implementable tasks to improve metrics coverage
in the OPNsense Prometheus exporter. Each task is self-contained with API
endpoint details, verified JSON response structures, proposed metric names, and
sample implementation code.

CURRENT COVERAGE STATUS (what we already collect):
  - Memory total/used/ARC (system collector)
  - Load average 1/5/15 (system collector)
  - Uptime, boot time, config change time (system collector)
  - Disk capacity per mount (system collector)
  - Swap total/used (system collector)
  - Temperature per sensor (temperature collector)
  - Interface traffic bytes/packets/errors/drops/multicasts (interfaces collector)
  - 50+ TCP/UDP/IP/ICMP/ARP/CARP/Pfsync protocol statistics (protocol collector)
  - Netisr dispatch stats, socket counts, route counts (network_diag collector)
  - PF states current/limit, per-interface pass/block packets/bytes (firewall)
  - Per-rule evaluations/packets/bytes/states (firewall_rules collector)
  - NTP per-peer stratum/delay/offset/jitter (ntp collector)
  - Mbuf current/cache/total/max/failures (mbuf collector)
  - System activity threads/CPU% (activity collector)
  - Firmware version/reboot/packages (firmware collector)
  - All service statuses (services collector)
  - ARP table, CARP VIPs, certificates, cron jobs
  - OpenVPN, WireGuard, IPsec, Unbound DNS, Dnsmasq, Kea DHCP

ARCHITECTURE REFERENCE:
- Adding a new collector requires 4 files (see CLAUDE.md "Adding a New Collector")
- Sub-collectors self-register via init() → collectorInstances slice
- Pattern to follow: internal/collector/dnsmasq.go + opnsense/dnsmasq.go (simplest example)
- All metrics use namespace "opnsense" and include "opnsense_instance" label
- Disable flags go in internal/options/collectors.go
- Endpoint URLs go in opnsense/client.go NewClient() endpoints map
- Subsystem name constants go in internal/collector/collector.go

This can be added to the EXISTING system collector (internal/collector/system.go)
since it's related system information. However, the CPU type endpoint is new and
needs to be registered.

Files to create/modify:
  - opnsense/system_resources.go    (add FetchSystemInfo and FetchCPUType methods)
  - internal/collector/system.go    (add systemInfo metric descriptor + emit)
  - opnsense/client.go              (add 2 endpoints)

Add to opnsense/client.go endpoints map:
  "cpuType":           "api/diagnostics/cpu_usage/getCPUType",
  "systemInformation": "api/diagnostics/system/system_information",

Notes:
  - The CPU type response is a JSON array: unmarshal into []string and take [0].
  - Parse the CPU string to extract cores/threads: look for pattern
    "(\d+) cores, (\d+) threads" using regex, or just expose the full string
    as cpu_model and skip cores/threads if parsing is fragile.
  - The versions array has 3 elements in fixed order: OPNsense, FreeBSD, OpenSSL.
    Parse "OPNsense " prefix off versions[0], "FreeBSD " off versions[1],
    "OpenSSL " off versions[2].
  - This is an INFO metric (always value 1) — use prometheus.GaugeValue.
  - These values rarely change, so they add minimal overhead per scrape.
  - This data is complementary to the firmware collector's opnsense_firmware_info
    metric. Consider whether to consolidate or keep separate. Recommend keeping
    separate since system_info includes FreeBSD and CPU data that firmware doesn't.
  - Treat failures on these endpoints as non-fatal (like memory.arc) — the rest
    of the system collector should still work even if these new calls fail.

          "store-buffer-length": 4028,

  opnsense_bpf_received_packets_total   counter labels: [process, interface, pid]
  opnsense_bpf_dropped_packets_total    counter labels: [process, interface, pid]
  opnsense_bpf_filter_packets_total     counter labels: [process, interface, pid]
  opnsense_bpf_store_buffer_bytes       gauge   labels: [process, interface, pid]
  opnsense_bpf_hold_buffer_bytes        gauge   labels: [process, interface, pid]
  opnsense_bpf_listeners_total          gauge   - count of bpf-entry array

IMPLEMENTATION:

Files to create/modify:
  - opnsense/bpf_statistics.go            (new — Fetch method + structs)
  - internal/collector/bpf.go             (new — collector)
  - opnsense/client.go                     (add endpoint)
  - internal/collector/collector.go       (add BPFSubsystem constant)
  - internal/options/collectors.go        (add disable flag)

Add to opnsense/client.go endpoints map:
  "bpfStatistics": "api/diagnostics/interface/get_bpf_statistics",

Notes:
  - Use "pid" as a label (converted to string) since the same process name
    may have multiple BPF listeners on different interfaces.
  - This is a LOW CARDINALITY collector — typically only 5-15 BPF listeners.
  - The main value is the dropped-packets counter: if this is non-zero and
    growing, a process is losing packets due to buffer overflow.
  - Consider if "pid" label is too volatile (PIDs change on restart). Could
    use only process+interface labels instead for more stable time series.
    RECOMMENDATION: Use process+interface only (skip pid) to avoid label churn.

NODE EXPORTER COVERAGE GAP ANALYSIS — CANNOT BE FILLED VIA API
================================================================================

The following node_exporter metrics CANNOT be replaced by this exporter because
the OPNsense API does not expose the underlying data. Users who need these
specific metrics will still need node_exporter or an alternative agent running
directly on the OPNsense box.

1. PER-CORE CPU TIME COUNTERS (node_cpu_seconds_total)
   - The OPNsense API only provides AGGREGATE CPU percentages via the activity
     endpoint (cpu user/nice/system/interrupt/idle).
   - The cpu_usage/stream endpoint provides per-core data but is a STREAMING
     endpoint that requires a persistent connection — not suitable for
     point-in-time Prometheus scrapes.
   - Impact: HIGH for users who need per-core CPU analysis.

2. MEMORY BREAKDOWN (node_memory_active/inactive/wired/laundry/free_bytes)
   - The API provides only total/used/ARC. FreeBSD's detailed vm page
     categories (active, inactive, wired, laundry, free) are not exposed.
   - The api/diagnostics/system/memory endpoint has malloc/zone stats but
     NOT the vm page category breakdown.
   - Impact: MEDIUM — total/used is sufficient for basic monitoring.

3. DISK I/O STATISTICS (node_disk_read/written_bytes_total)
   - No OPNsense API endpoint provides disk read/write throughput, IOPS,
     or I/O latency. Only filesystem capacity (df-style) is available.
   - Impact: MEDIUM — disk I/O is less critical for firewall appliances.

4. KERNEL COUNTERS (context switches, interrupts, forks, page faults)
   - These low-level FreeBSD kernel counters (from sysctl) are not exposed
     through any OPNsense API endpoint.
   - Impact: LOW — rarely needed for firewall monitoring.

5. FILE DESCRIPTORS / OPEN FILES
   - FreeBSD's kern.openfiles / kern.maxfiles are not exposed via API.
   - Impact: LOW — file descriptor exhaustion is rare on firewalls.

NODE EXPORTER COVERAGE GAP ANALYSIS — CANNOT BE FILLED VIA API
================================================================================

The following node_exporter metrics CANNOT be replaced by this exporter because
the OPNsense API does not expose the underlying data. Users who need these
specific metrics will still need node_exporter or an alternative agent running
directly on the OPNsense box.

1. PER-CORE CPU TIME COUNTERS (node_cpu_seconds_total)
   - The OPNsense API only provides AGGREGATE CPU percentages via the activity
     endpoint (cpu user/nice/system/interrupt/idle).
   - The cpu_usage/stream endpoint provides per-core data but is a STREAMING
     endpoint that requires a persistent connection — not suitable for
     point-in-time Prometheus scrapes.
   - Impact: HIGH for users who need per-core CPU analysis.

2. MEMORY BREAKDOWN (node_memory_active/inactive/wired/laundry/free_bytes)
   - The API provides only total/used/ARC. FreeBSD's detailed vm page
     categories (active, inactive, wired, laundry, free) are not exposed.
   - The api/diagnostics/system/memory endpoint has malloc/zone stats but
     NOT the vm page category breakdown.
   - Impact: MEDIUM — total/used is sufficient for basic monitoring.

3. DISK I/O STATISTICS (node_disk_read/written_bytes_total)
   - No OPNsense API endpoint provides disk read/write throughput, IOPS,
     or I/O latency. Only filesystem capacity (df-style) is available.
   - However, SMART provides Total_LBAs_Written (id=241) which gives
     cumulative bytes written — a partial substitute.
   - Impact: MEDIUM — disk I/O is less critical for firewall appliances.

4. KERNEL COUNTERS (context switches, interrupts, forks, page faults)
   - These low-level FreeBSD kernel counters (from sysctl) are not exposed
     through any OPNsense API endpoint.
   - Impact: LOW — rarely needed for firewall monitoring.

5. FILE DESCRIPTORS / OPEN FILES
   - FreeBSD's kern.openfiles / kern.maxfiles are not exposed via API.
   - Impact: LOW — file descriptor exhaustion is rare on firewalls.

NOTE ON ZENARMOR: Zenarmor (os-sensei) is installed on the test instance but
does NOT expose a standard OPNsense REST API. It uses a proprietary separate
API with its own authentication and requires a paid business license. Building
a collector for Zenarmor is NOT feasible through the standard OPNsense API.

CONCLUSION: With the existing collectors plus TODOs 20-31 above, the OPNsense
exporter will cover approximately 90%+ of what node_exporter provides for
FreeBSD systems, PLUS extensive firewall, VPN, security, and plugin coverage
that node_exporter cannot provide at all. The remaining gaps (per-core CPU,
memory page categories, disk I/O) require direct system access.

For most firewall monitoring use cases, the OPNsense exporter provides
SUPERIOR coverage in areas that matter most: network interfaces, firewall
state tables, protocol statistics, NTP, temperature, VPN metrics, disk health,
threat intelligence, and security — none of which node_exporter handles as
comprehensively.
